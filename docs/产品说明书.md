## 1. 产品定位

### 1.1 一句话

**Reverie 是把一句灵感变成一部可玩的互动影游的平台**——AI 负责规模化生产，树状剧情框架负责收束分支，人在关键节点做判断与精修。

### 1.2 我们要解决的问题

| 痛点 | 传统做法 | Reverie 方案 |
|---|---|---|
| 互动叙事创作门槛高 | 编剧 + 分支策划 + 美术 + 程序分工，周期以月计 | 一句构想 → AI 多 Agent 流水线粗生成 → 人审改 |
| 分支爆炸 | 手写每条支线，组合数指数增长 | **锚点收束**：局部可展开，主线在关键交互节点汇合 |
| 内容荒 | 依赖少数专业团队，产能有限 | UGC + AI 粗生成，平台日产可达百部量级（目标） |
| 沉浸感断裂 | 多模态素材风格不一 | Story Package 统一 `visualStyle` + 角色 sheet + 场景库 + 版本化资产 |
| 玩家「选了没用」 | 假选择、装饰性分支 | 剧作法 QA + 逻辑校验，选项必须改写关系/信息/处境 |

## 2. 核心机制：锚点树 + AI 填充 + 逻辑校验

这是 Reverie 与「纯 AI 聊天叙事」的本质区别，也是支撑**规模生产**的产品内核。

### 2.1 设计原则

```
                    ┌─────────────────────────────────────┐
                    │  创作者定骨架：大纲 / 锚点 / 结局矩阵   │
                    └──────────────────┬──────────────────┘
                                       ▼
              ┌────────────────────────────────────────────┐
              │  锚点树（Anchor Tree）                       │
              │  · 关键交互节点 = 预制 beats + 选项 + 分镜   │
              │  · 局部支线可在两锚点间展开（AI 填充）        │
              │  · 每幕末必须收束到下一锚点（禁止无限发散）      │
              └──────────────────┬─────────────────────────┘
                                 ▼
              ┌────────────────────────────────────────────┐
              │  运行时                                       │
              │  · 锚点内：播放预制内容（快、稳、可控）          │
              │  · 锚点间：next=null / 自由输入 → AI 流式续写   │
              │  · 续写结束：逻辑校验 → 收敛回下一锚点           │
              └────────────────────────────────────────────┘
```

**玩家感受**：选择在锚点处「真的改了故事」；在锚点间自由发挥时，AI 补全细节但仍回到主线。

**创作者感受**：不必手写每种可能性，只需设计锚点、选项路由、结局矩阵与圣经约束。

### 2.2 锚点树结构定义

| 概念 | 说明 | 在 Story Package 中的映射 |
|---|---|---|
| **锚点节点（Anchor）** | 关键交互点：预制剧本 + 2~3 两难选项 + 分镜 | `StoryNode`（`choices[].next` 指向下一锚点或结局） |
| **填充区（Fill Zone）** | 两锚点之间允许 AI 实时续写的区间 | `PackageChoice.next = null` 或自由输入触发 `/api/generate` |
| **收束点（Convergence）** | 多条支线路由必须汇合的锚点 id | 节点规划中显式标注；QA 检查可达性 |
| **结局节点（Ending）** | 终止分支，无选项 | `isEnding: true`，`endingLabel` 供收集 |
| **付费高光（Premium）** | 高风险高回报选项 | `choices[].premium: true`，消耗钻石 |

### 2.3 剧作法约束（全局注入）

所有生成 Agent 与运行时续写均注入 `SCREENWRITING_CRAFT`（见 `lib/storyPackage.ts`）：

- 钩子前置、戏剧冲突、先虐后爽、反转伏笔、情绪过山车
- 选项必须两难，禁止假选择（多选项同 next 且无差异）
- 每幕结束留 cliffhanger

### 2.4 逻辑校验（三层）

| 层级 | 时机 | 检查内容 | 实现 |
|---|---|---|---|
| **L1 脚本质检** | 大纲→脚本后 | 场景格式、路由语法、结局覆盖、台词长度 | `lib/scriptValidator.ts`（规划） |
| **L2 包结构 QA** | 导入 / 发布前 | 断链、孤儿节点、缺结局、缺 low/payoff 节拍 | `validatePackage()` ✅ 已有 |
| **L3 运行时校验** | AI 续写后 | 人设不跑偏、不推翻已发生事实、必须朝下一锚点推进 | `streamScenePkg()` 约束 + 收束检测（规划） |

## 3. 全链路自动化生产（Studio 核心）

目标：**一句构想 → 可发布 Story Package**，中间产物均可人工审改。

### 3.1 生产流水线总览

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ ① 一句构想    │ →  │ ② 剧本大纲    │ →  │ ③ 游戏脚本    │ →  │ ④ 故事包      │ →  │ ⑤ 视觉资产    │
│ idea         │    │ outline       │    │ game script  │    │ Story Package│    │ images/video │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
     Agent               Agent               Agent            importScript         批量生图
  expandIdea()      (已有 analyze)      write+validate      parseScriptTree      /api/image
```

### 3.2 各阶段规格

#### 阶段 ① 一句构想 → 剧本大纲

| 项 | 内容 |
|---|---|
| **输入** | 1~3 句话 + 可选题材/尺度 |
| **输出** | 完整故事大纲 Markdown（含叙事模型、完整剧情梗概、分幕详述、关键节点、结局矩阵） |
| **必含区块** | Logline、基本设定、叙事模型、人物（欲望+秘密）、完整剧情梗概、分幕剧情、关键交互节点、结局矩阵、伏笔与回收、情绪曲线 |
| **落盘** | `data/outlines/<id>.md` |
| **Agent** | `lib/ideaAgent.ts` |
| **现状** | 已实现：一句构想 → 叙事模型选择 → 大纲 Markdown |

#### 阶段 ② 剧本大纲 → 游戏脚本

| 项 | 内容 |
|---|---|
| **输入** | 大纲 Markdown |
| **输出** | 可导入 plain-text 游戏脚本（场景、对白、关键选择、明确路由、结局分支） |
| **Agent 三步** | ① `analyzeOutline` ② `planStoryNodes` ③ `writeGameScript` |
| **规划增强** | ④ `validateScript` ⑤ `repairScript`（待实现） |
| **落盘** | `data/scripts/<id>.txt` + `.meta.json` |
| **实现** | `lib/scriptAgent.ts` ✅ 骨架已有，质检/按场写待加强 |

#### 阶段 ③ 游戏脚本 → Story Package

| 项 | 内容 |
|---|---|
| **输入** | 游戏脚本全文 |
| **输出** | `StoryPackage` JSON |
| **两步** | `parseScriptMeta`（圣经/角色/场景）+ `parseScriptTree`（节点/选项/结局） |
| **落盘** | `data/stories/<id>.json`，`status: draft` |
| **实现** | `lib/storyTree.ts` → `importScript()` ✅ |

#### 阶段 ④ 故事包 → 视觉资产

| 项 | 内容 |
|---|---|
| **输入** | 各节点 `imagePrompt` + 包级 `visualStyle` + 角色 `sheet` |
| **输出** | 节点 `asset` URL、场景库 `asset`、海报 `poster` |
| **模型** | Seedream 文生图（`ARK_IMAGE_MODEL`） |
| **约束** | `IMAGE_BUDGET` 防超支；缺图回退 location / 海报 |
| **实现** | `/api/image` + 播放器实时生图 ✅；批量预渲染脚本 `scripts/gen-assets.mjs` 部分 |

#### 阶段 ⑤ 人审 → 发布

| 项 | 内容 |
|---|---|
| **编辑器** | `/admin/[id]` 六 Tab 精修 |
| **QA 门禁** | `validatePackage()` 有 error 禁止发布 |
| **状态机** | `draft → review → published → archived` |
| **实现** | ✅ |

### 3.3 创作者可选路径（Studio 入口）

| 路径 | 适合 | 步骤 |
|---|---|---|
| **极速** | 快速验证 | 一句构想 →（跳过纲）→ 故事包 |
| **标准** | 质量可控 | 一句构想 → 大纲 → 脚本预览 → 故事包 |

### 6.2 核心数据模型：Story Package

单一可序列化单元，前后台共用（`lib/storyPackage.ts`）：

```
StoryPackage
├── 元信息：id, title, tagline, genre, ageRating, status, version
├── 创作溯源：idea（原始构想）
├── 视觉统一：visualStyle, poster
├── 故事圣经 arc：premise, hook, payoff, beatSheet, coreConflict …
├── 角色 characters[]：name, sheet, persona, voice, ref
├── 场景 locations[]：key, name, prompt, asset
└── 锚点树 nodes[]
    ├── beats[]：旁白/对白
    ├── beatFunction：hook|setup|conflict|twist|low|payoff|climax|ending
    ├── cliffhanger：幕末悬念
    ├── imagePrompt / asset：分镜
    ├── choices[]：label, next(锚点id|null), premium, cost
    └── isEnding / endingLabel
```

**`next` 语义**：

- `string` → 跳转预制锚点（确定性分支）
- `null` → 进入填充区，由 `/api/generate` AI 续写，完成后收敛回 `actAnchorNode()`

### 6.3 AI Agent 编排

| Agent | 文件 | 输入 → 输出 | 状态 |
|---|---|---|---|
| 构想扩纲 | `lib/ideaAgent.ts`（规划） | idea → outline.md | 待做 |
| 大纲分析 | `scriptAgent.analyzeOutline` | outline → OutlineAnalysis JSON | ✅ |
| 节点规划 | `scriptAgent.planStoryNodes` | analysis → NodePlan（锚点树） | ✅ |
| 脚本撰写 | `scriptAgent.writeGameScript` | plan → game script.txt | ✅ 待加强为按场写 |
| 脚本修复 | `scriptAgent.repairScript`（规划） | script + errors → fixed script | 待做 |
| 包元信息 | `storyTree.parseScriptMeta` | script → 圣经/角色/场景 | ✅ |
| 包拆树 | `storyTree.parseScriptTree` | script → nodes[] | ✅ |
| 构想直达 | `storyTree.generatePackage` | idea → StoryPackage | ✅ |
| 运行时续写 | `ai.streamScenePkg` | 上下文+选择 → 流式 beats | ✅ |

**调用方式**：全部通过 `chat()` / SSE，OpenAI 兼容；无 key 走 Mock 兜底。

### 6.4 运行时播放逻辑

```
加载 StoryPackage
  → getStartNode()
  → 渲染 beats + 背景图
  → 展示 choices
      ├─ next = 节点id  → 跳转预制锚点
      ├─ next = null     → POST /api/generate（SSE 流式）
      │                    └─ 收敛检测 → actAnchorNode 或结局
      └─ premium         → 扣钻石后执行
```

续写 Prompt 注入：故事圣经 + 当前幕 + 已发生摘要 + 剧作法 + 「必须推进到下一锚点」。

### 6.5 视觉生产方案

| 阶段 | 策略 | 说明 |
|---|---|---|
| **创作时** | 每节点 `imagePrompt`（英文）+ 包级 `visualStyle` + 角色 `sheet` | LLM 生成 |
| **预览时** | 播放器调用 `/api/image` 实时生图 | 受 `IMAGE_BUDGET` 限制 |
| **发布前** | 批量预渲染写入 `node.asset` | 避免玩家等待 |
| **分发** | [规划] 上传 COS，`release/{hash}/` 版本路径 | 对齐 Castloop CDN 模式 |
| **视频** | [规划] Seedance `seedance-*`，关键锚点预生成，填充区仍用图 | 成本可控 |

一致性三板斧：

1. 所有 prompt 拼接同一 `visualStyle`
2. 角色引用 `sheet` + 可选 `ref` 参考图
3. 场景复用 `locations[].asset`

| **Alpha** | PostgreSQL + S3 兼容存储 | 同上 + 后台生图队列 | 小范围内测 |
| **Beta** | 分库：内容库 / 用户库 | Worker 处理 Agent 长任务 | UGC 开放 |
| **生产** | COS + CDN + 版本 hash | 独立 Agent 服务 + GPU 生图/视频池 | 对标 Castloop 规模 |

---

## 7. 游戏脚本生产标准（纲→脚本 Agent 产出规范）

Agent 产出必须对齐平台统一的可导入脚本格式，方可无损 `importScript()`。

### 7.1 文件结构

```
片名 / 题材 / 尺度 / 人物
场景 1 [标题]
  [旁白] …
  角色名：…
关键选择 1
  A … (→ 场景 2)
  B … (→ 场景 2，好感-1)
  C 💎 … (→ 场景 2 或 结局·XXX)
…
结局分支
  结局名 —— 收束正文（2~4句）
```

### 7.2 质量门禁

| 检查项 | 标准 |
|---|---|
| 场景数 | 4~6 |
| 结局数 | 2~4，各有独立触发路由 |
| 选项 | 每处 2~3 个，不得全同 next |
| 💎 | 1~2 处 |
| 节拍 | 含 hook、low/twist、payoff/climax |
| 台词 | 单条 ≤60 字；第二人称「你」 |
| 路由 | 明确 `(→ 场景 N)` 或 `(→ 结局·名)` |

## 11. 总结

Reverie 的核心竞争力不是「AI 能写故事」，而是：

1. **树状标准化框架**——锚点预制 + 局部 AI 填充 + 强制收束，兼顾玩家能动性与产能可控；
2. **全链路 Studio**——从一句灵感到可玩影游，AI 粗生成、人审改、QA 门禁；
3. **Story Package 单一真相源**——剧本、分支、视觉、版本一体化，支撑万级内容规模；
4. **多模态一致性的工程化**——`visualStyle` + 角色 sheet + 资产版本化，解决「每张图不像同一部戏」。

Castloop 证明了市场；Reverie 用开源可复刻的技术栈 + 显式锚点树机制，把「创作端自动化」和「消费端体验」放在同一个产品里迭代。
