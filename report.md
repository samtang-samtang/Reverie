## 学习/探索
- 借鉴 CastLoop 的「树状标准化剧情框架」思路：主线收拢到关键交互节点、支线局部展开、AI 填充节点间内容并做逻辑校验，以此平衡「玩家选择自由」与「内容不无限发散」。

## 今日问题与解决
- **问题**：AI 扩纲/剧本解析反复失败，报 `Expected ',' or ']' ... in JSON`（位置 6921 / 8970 等），导致整次生成不可用。
  **解决**：定位为 LLM 输出 JSON 语法不稳定，转向更鲁棒的解析（正则修复/容错解析），减少硬失败。（仍在持续加固，见明日计划）
- **问题**：单一环节出错就让整条流程全盘不可看，已生成成功的内容也被丢弃；且兜底用的本地预置大纲与创作者输入不一致，等于交付无效版本。
  **解决**：改为部分成功也可落库/查看，避免无意义兜底。
- **问题**：人物图生成结果杂乱，不符合「聊天人物形象照」的产品诉求；图片/视频出现中文、生成 API 单次点击却重复调用。
  **解决**：明确人物图统一为形象照规范（沉淀为后续所有剧本通用约束），资产生成禁用中文，排查并修复重复调用。

## 明日重点计划
- 彻底根治扩纲/解析的 JSON 稳定性问题，确保「不报错、可复现」，并保证任意一句话构想都产出完整长篇故事大纲（含全部剧情、三幕详述）。

- 完善资产管理与后续视频生成方案。
